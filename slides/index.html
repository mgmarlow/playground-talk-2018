Thinking in React

  1. Start with a mock
  2. Break the UI into a component hierarchy
  3. Build a static version (HTML)
  4. Identify and add state (logic and data)
slow down
  • Picture completed UI, break it into pieces (components)
  • Code the markup of each component (stateless components)
  • Think about interactions, logic
  • Code interactions, logic (alter HTML as needed)

The Mocking Phase

  • Each box is its own component, color-coded
  • You can see how intuitive this method of development is
  • Great approach, but has issues
  • The real world of UI development looks more like this picture
  • CDD painpoint: When the customer inevitably changes their mind, having all of that markup developed is just more code to be rewritten
  • The key is to have a demonstrable product at every point in the dev cycle

Scenario-driven Development

  1. Start with a mock
  2. Break the UI into a component hierarchy
  3. Build a complete, stateful version of the simplest component
  4. Repeat #3 until entire UI is finished
  • Steps 1 and 2 are the same as CDD
  • As components are developed, they are completed with all of their variations accounted for
  • The difference is at every step in the process, you have a demonstrable component

Scenario-driven Development

  • Fully demo-able UI at every step in development
  • Changes are easier, affect fewer components
  • More isolated, functional components
  • Problem: How can we work on a piece of the UI without having the entire thing structured?
